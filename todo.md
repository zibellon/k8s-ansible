## Структура установки HELM
Разделить все установки на три шага. Для каждого шага - отдельный helm-chart. Но все три шага в одном playbook
- pre. Вот тут ставим все Network-policy. И мб что-то еще
- install. Вот тут ставим основной компонент. Вот этот helm-chart будет не у всех. Например у cilium его не будет, так как основная установка производится из официального helm.
- post. Вот тут ставим что-то, что ставится только после установки основного компонента (для cilium, в этом шаге ставим CiliumClusterwideNetworkPolicy - можно сделать тол ко после установки crds)

Ansible

- политики для vault. Полностью перенести в playbook. Удалить task. Теперь они запускаются отдельно - и таск не нужен

- конфигурация vault (вся дурка с ключами и так далее), перенести в отдельный playbook. vaulta?config.yaml. вызывается отдельно, указать в инструкции. То есть установка vault === именно запуск контейнеров

- gitlab. В pre стадии должны остаться только Network-policy + ESO. Все остальное в стадию install. Именно перенести шаги по стадиям

- gitlab. Конфигурация (создание пользователей, PAT, мб даже группы и репозитории) - перенести в отдельный gitlab-config.yaml

- argocd. Конфигурацию (пароль админа - только он), вынести в отдельный playbook

- argocd-git-ops. appProject + application = теперь это массив из объектов. Там есть все необходимые параметры для создания двух этих сущностей. 1 appProject === 1 application. То есть: вот сколько надо - столько и загоняем

- argocd-git-ops. ESO для подключения к нужным репозитория - создаётся сразу, при установке ArgoCd. Там можно указать repo-creds или другой тип (второй, я забыл как он называется)

- argocd-git-ops. Генерация ключей. Отдельный массив argocd_gitops_repo_creds. Проверка на уникальность, проверка что каждый элемент имеет пересечение по vaultpath (из ESO integration).

- argocd-git-ops. Автоматическое создание репозитория в gitlab (internal)

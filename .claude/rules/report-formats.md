# Report Formats & Defensive-Additions Rules

Единый источник истины для трёх вещей:

1. Формат отчётов DevOps / DevOps-docs → TeamLead (§1).
2. Запрет «защитных» добавок в коде (§2).
3. Запрет «защитных» добавок в документации (§3).

Все три темы раньше дублировались в `.claude/prompts/*-bootstrap.md` и [`team-workflow.md`](team-workflow.md). Теперь они только здесь — bootstrap-файлы и team-workflow.md ссылаются на этот файл.

---

## 1. Формат отчётов

Исполнитель (DevOps или DevOps-docs) возвращает user'у блок текста в одном из трёх форматов. User копирует его в Opus chat TeamLead-у.

Статус один из: `DONE`, `BLOCKED`, `NEEDS_CLARIFICATION`.

### 1.1 DONE

```markdown
# Status: DONE
# Task: SUB-N

## Files changed
- <путь> (lines X–Y): <что сделано — одним предложением>
- <путь>: created / modified / deleted

## Verification
- <что проверил + результат>
  (например: ansible-playbook -i hosts-vars/ -i hosts-vars-override/ <file> --syntax-check → exit 0)
- <...>

## Side issues (не исправлено)
- <если заметил рядом проблему — перечисли, без исправления>
- или "нет"
```

**Строгое правило — `Files changed`:** одна запись на КАЖДЫЙ изменённый файл. Сводные формулировки типа «все N файлов отредактированы», «17 файлов приведены к единообразию», «остальное сделано в предыдущей сессии» **не принимаются**. TeamLead верифицирует каждый файл отдельно — для этого ему нужен явный список. Если файлов много (>10) — всё равно перечисляй все, одну строку на файл.

### 1.2 BLOCKED

```markdown
# Status: BLOCKED
# Task: SUB-N

## Reason
<что конкретно блокирует: отсутствует файл / нет доступа / зависимая задача не сделана / противоречие в спеке>

## Needed to unblock
<что нужно для разблокировки: разрешение / зависимость / доступ / решение>

## Partial work
<что уже успел сделать, если что-то сделал, или "ничего">
```

### 1.3 NEEDS_CLARIFICATION

```markdown
# Status: NEEDS_CLARIFICATION
# Task: SUB-N

## Conflict
<конкретный пункт спеки, который неясен или противоречит реальному состоянию файла>

## Options
a) <вариант решения>
b) <вариант решения>
c) <вариант решения>

## My recommendation
<a|b|c> — <обоснование>
```

### 1.4 Общие правила для всех трёх форматов

- Заголовок строго `# Status: <DONE|BLOCKED|NEEDS_CLARIFICATION>` и `# Task: SUB-N` — TeamLead парсит глазами, отклонения от формата усложняют verify.
- Не добавлять секции, не указанные в шаблоне. Если есть что сказать сверх — положи в `Side issues` (для DONE) или в свободный комментарий после блока.
- Не обобщать. Конкретика путей, команд, строк в `Verification`.

## 2. Запрет защитных добавок — для кода

Применимо к DevOps при работе над playbooks, task includes, charts, vars files.

### 2.1 Запрещено без явного указания в SUB-спеке

- `failed_when`, `ignore_errors`, `any_errors_fatal`
- `tags: [...]` (исключение: `tags: [always]` на assert-блоке task include по Rule 19 `playbook-conventions.md`)
- `become`, `environment`, `no_log`
- `check_mode`, `diff`
- Дополнительные `register` / `set_fact`, не указанные в плане
- Блоки `rescue` / `always`
- Дополнительные `debug`-задачи, не указанные в плане
- Комментарии-пояснения «зачем это нужно» (если спека не требует)
- «Защитные» проверки вида «а вдруг файл уже есть / а вдруг переменная пустая» — если спека не требует, не добавлять

### 2.2 Процедура если считаешь нужным

STOP → отчёт `NEEDS_CLARIFICATION` с конкретным предложением и обоснованием («предлагаю добавить `ignore_errors: true` на задачу X потому что Y»). Жди решения TeamLead.

**Никаких «я добавил на всякий случай, это же лучше».** Spec — закон. Отклонение от spec = эскалация, не импровизация.

## 3. Запрет защитных добавок — для документации

Применимо к DevOps-docs при работе над `.claude/rules/*.md`, `CLAUDE.md`, `todo.md`, комментариями в коде.

### 3.1 Запрещено без явного указания в SUB-спеке

- Новые подразделы, которых нет в плане
- Примеры, которых нет в плане (даже если «так понятнее»)
- `Warnings` / `Notes` / `Gotchas` боксы, не упомянутые в задаче
- Изменения в соседних записях каталога «для единообразия»
- Переформулирование существующих фрагментов (если не сказано явно «переписать»)
- `TODO` / `FIXME` пометки
- «Обучающие» вводные в новые записи, которых нет у соседей
- Перенумерация старых записей без указания (если добавляешь § посередине — только если спека говорит)

### 3.2 Процедура если считаешь нужным

Аналогично §2.2: STOP → `NEEDS_CLARIFICATION` с конкретным предложением. Жди решения.

### 3.3 Правила стиля (всегда действуют, не требуют указания)

1. **Формат совпадает с соседями.** Если в файле записи оформлены `### X.Y <название>` с bullets `- **Purpose.**`, `- **Input.**` — новая запись точно в том же формате, в правильной позиции по нумерации.
2. **Язык** — как в соседних записях того же файла. Русский в пояснениях, английский в формальных таблицах — оба встречаются. Не меняй язык соседей.
3. **Краткость.** Документация — справочник, не учебник. Не дублируй содержимое других файлов — ставь ссылку.
4. **Ссылки на файлы.** Относительные пути в бэктиках: `` `playbook-app/tasks/tasks-foo.yaml` ``. Для файлов в `.claude/rules/` — markdown-ссылка: `[имя](имя.md)`.
5. **Счётчики.** Если в документе есть числовой счётчик элементов (`(20 tasks)`, `20 reusable task includes`), при добавлении/удалении элемента обновить счётчик.

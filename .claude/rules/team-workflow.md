# Team Workflow — Manual Chat Mode

Единый rulebook для ручного multi-chat workflow. В отличие от автоматизированного agent-team (который оказался хрупким из-за Ink-багов и сложности восстановления), здесь user вручную переносит задачи и отчёты между двумя chat-окнами.

**Режим работы:** manual chat mode.
**Агентов как процессов нет.** Есть только user и два типа chat-окон: Opus-chat (TeamLead) и Sonnet-chat (DevOps или DevOps-docs).

---

## 1. Команда и роли

| Роль | Модель | Где живёт | Что делает |
|---|---|---|---|
| **User** | — | — | Приносит идею, утверждает план, копирует промпты/отчёты между chat-окнами, принимает финальный результат |
| **TeamLead** | Opus 4.7 | Chat-окно №1 | Обсуждает идею с user, задаёт уточняющие вопросы, декомпозирует на SUB-tasks, генерирует промпт для DevOps, верифицирует отчёты, коммитит результат |
| **DevOps** | Sonnet 4.6 | Chat-окно №2 | Код: playbooks, charts, task includes, vars |
| **DevOps-docs** | Sonnet 4.6 | Chat-окно №3 (при необходимости) | Документация: `.claude/rules/*`, `todo.md`, комментарии, `hosts-extra.example.yaml` |

**Границы между ролями жёсткие:**

- TeamLead не пишет код и не пишет доку — только координирует и верифицирует.
- DevOps не пишет доку — только код.
- DevOps-docs не пишет код — только доку.
- User — единственный канал передачи данных между chat-окнами.

---

## 2. Жизненный цикл задачи — 10 шагов

```
┌─────────────────────────────────────────────────────────┐
│ 1.  User → TeamLead: "идея такая..."                    │
│ 2.  TeamLead: уточняющие вопросы (если нужно)           │
│ 3.  User: ответы                                        │
│ 4.  TeamLead: план с SUB-tasks (N штук, с порядком)     │
│ 5.  User: approve (или правки → в 4)                    │
│                                                          │
│  ДЛЯ КАЖДОЙ SUB-N:                                      │
│ 6.  User → TeamLead: "дай промпт для SUB-N"             │
│ 7.  TeamLead: промпт в формате §4 (копируемый)          │
│ 8.  User копирует → вставляет в Sonnet chat             │
│ 9.  DevOps выполняет → отдаёт отчёт в формате §5        │
│ 10. User копирует отчёт → вставляет в Opus chat         │
│ 11. TeamLead verify (§6) → commit (§7) или доработка    │
│                                                          │
│  Когда все SUB-N закрыты:                               │
│ 12. TeamLead: финальный сводный отчёт user              │
└─────────────────────────────────────────────────────────┘
```

**Принцип последовательности:** следующая SUB-task не начинается пока текущая не закрыта (verify + commit). Параллелизма нет — это осознанный выбор в пользу простоты и контроля.

## 3. Cold-start протокол

### 3.1 Для TeamLead (Opus chat)

При открытии нового chat-окна с Opus 4.7 — user вставляет содержимое `.claude/prompts/teamlead-cold-start.md` как первое сообщение.

TeamLead:
1. Читает CLAUDE.md (карта проекта)
2. Читает этот файл (`team-workflow.md`) — workflow и правила
3. Пробегает `.claude/prompts/devops-bootstrap.md` — знать границы DevOps, чтобы давать корректные спеки
4. Подтверждает одним сообщением: понял workflow, прочитал инварианты CLAUDE.md §0, знает границы исполнителя
5. Спрашивает user: какую задачу будем делать

### 3.2 Для DevOps (Sonnet chat)

При открытии нового chat-окна с Sonnet 4.6 — user вставляет содержимое `.claude/prompts/devops-bootstrap.md` как первое сообщение.

DevOps:
1. Читает CLAUDE.md
2. Читает релевантные `.claude/rules/*.md` (playbook-conventions.md обязательно)
3. Подтверждает готовность
4. Ждёт от user первую SUB-task

**Правило:** bootstrap вставляется **один раз на сессию chat-окна**. Далее в этом же окне user вставляет только task-спеки (без повторения bootstrap). Если chat-окно закрыто и открыто заново — bootstrap надо вставить снова.

### 3.3 Для DevOps-docs (Sonnet chat)

Аналогично §3.2, но bootstrap файл `.claude/prompts/devops-docs-bootstrap.md`.

**Когда создавать отдельное DevOps-docs chat-окно:**

- Если в плане есть SUB-task с изменениями в `.claude/rules/*`, `CLAUDE.md`, `todo.md`, `hosts-extra.example.yaml`, комментариях.
- Если SUB-task смешанная (код + доку) — TeamLead должен разбить на две отдельные SUB: одну для DevOps, одну для DevOps-docs.

## 4. Формат SUB-task промпта (TeamLead → user → DevOps)

TeamLead отдаёт user блок текста, готовый к копированию-вставке в Sonnet chat.

**Требования к спеке:**

- **Самодостаточность** — исполнитель не должен догадываться. Если правка зависит от текущего содержимого файла — TeamLead сам проверяет через Read и включает контекст в спеку.
- **Точность** — номера строк, дословный текст, якоря (заголовки, уникальные строки).
- **Никаких "на усмотрение"** — только однозначные инструкции.
- **Verify — обязательно** — конкретная команда, по результату которой исполнитель может отличить успех от провала.

### 4.1 Skeleton — для копирования

Копируй блок ниже целиком и заполняй секции. Это канонический шаблон, от него не отклоняться.

````markdown
# SUB-N: <короткий заголовок>

## Цель
<одно предложение — чего добиваемся>

## Файл(ы)
- <абсолютный или относительный путь>
- <...>

## Референс (если есть)
- <файл-образец и что оттуда брать — формат, стиль, структура>

## Изменения

### В файле <путь>:

**Что:** <точное описание правки — что меняется, куда вставляется, что удаляется>

**Позиция:** <после строки X, перед секцией Y — точно>

**Новый текст** (если вставка/замена):
```<lang>
<точное содержимое>
```

## Что НЕ делать
- <явный список запретов для этой конкретной SUB>
- <...>

## Verify перед DONE
1. <конкретная команда или действие + ожидаемый результат>
2. <...>

## Отчёт
Пришли обратно TeamLead по формату [`report-formats.md`](report-formats.md) §1 (DONE / BLOCKED / NEEDS_CLARIFICATION).
````

---

## 5. Формат отчёта (DevOps → user → TeamLead)

Канонические форматы `DONE` / `BLOCKED` / `NEEDS_CLARIFICATION` — в [`report-formats.md`](report-formats.md) §1. Дублировать их здесь не нужно.

**Строгое правило для секции `Files changed` (DONE):**

- Одна запись — один файл. Всегда.
- Сводные формулировки типа «все 17 файлов отредактированы», «остальное сделано в предыдущей сессии» — **не принимаются**.
- TeamLead верифицирует каждый изменённый файл отдельно, для этого ему нужен явный и полный список. Если файлов >10 — всё равно перечисляй все, строка на файл.

## 6. Verify-протокол (обязанность TeamLead)

После получения `REPORT` от user — TeamLead НЕ закрывает SUB-task, пока не проверил работу сам через Read/Bash/Grep.

| Тип артефакта | Как verify |
|---|---|
| Новый playbook / task include | `Read` + `ansible-playbook --syntax-check` через временный stub |
| Chart template | `Read` + (при наличии кластера) `helm template` dry-run |
| `vars` файл | `Read` + `python3 -c 'import yaml; yaml.safe_load(open("<file>"))'` |
| Изменение в `.claude/rules/*` / CLAUDE.md | `Read` изменённых строк + `Grep` на целостность ссылок/якорей |
| Правки в существующем файле | `git diff <file>` + syntax-check если применимо |

**Если verify выявил расхождение с планом** — TeamLead возвращает user точный список правок для следующей итерации DevOps (не "исправь стиль", а "удалить строку 32, строки 36 переписать как X").

**Лимит итераций доработки:** один возврат на SUB-task. После второго провала verify — TeamLead останавливает SUB и эскалирует user (см. §9).

## 7. Commit-протокол

### 7.1 Когда коммитить

После успешного verify каждой SUB-task. Не ждать всех SUB — коммитить атомарно по мере завершения. Это даёт чистую историю + возможность откатиться посредине.

### 7.2 Формат commit-message

```
<type>: <короткий заголовок — что SUB сделала>

<подробное описание — зачем, со ссылкой на SUB-task если важно>

- файл1: что изменилось
- файл2: что изменилось

Part of: <описание общей задачи, если это часть большого плана>
```

**Типы (conventional commits):**

- `feat` — новая функциональность
- `fix` — исправление бага
- `refactor` — переработка без изменения поведения
- `docs` — только документация
- `chore` — служебные изменения

### 7.3 Что коммитить

- Только файлы, изменённые в рамках SUB-task
- **Никогда** не включать `hosts-vars-override/` (secrets)
- **Никогда** не использовать `git add -A` / `git add .` вслепую — всегда конкретные файлы по имени

### 7.4 Что НЕ делать

- Не пушить в remote без явного запроса user
- Не использовать `--no-verify` или `--amend` без явного запроса
- Не делать `git reset --hard`, `git push --force`, `git clean -f` без явного запроса
- Не коммитить если verify провалился

## 8. TeamLead self-discipline

### 8.1 Уточнения с явными вариантами

Перед approval плана — задавать уточняющие вопросы. Каждый вопрос с явными вариантами (a/b/c). **Не говорить** "если не ответишь — беру дефолты" — это создаёт двусмысленность. Ждать явного ответа на каждый пункт.

### 8.2 Самодостаточные спеки

SUB-task промпт должен быть полон. User не должен разбираться "а что там DevOps увидит в файле" — TeamLead делает Read сам и включает релевантный контекст.

### 8.3 Verify обязателен

После DONE-отчёта — всегда Read / Bash / Grep для проверки. Не верить отчёту на слово. Это правило не обсуждается.

### 8.4 Границы самовыполнения

TeamLead **делает сам** (не делегирует user'у / исполнителю):

- Обсуждение плана с user
- Уточняющие вопросы и approval
- Генерация SUB-task промптов
- Verify результатов (Read, Grep, Bash)
- Commit после успешного verify
- Финальный сводный отчёт user
- Правки, оставшиеся от исполнителя после 2 провалов verify (принцип: 1 итерация возврата, потом — эскалация user)

TeamLead **делегирует** (не делает сам):

- Создание / правки кода (playbooks, charts, task includes, vars) → DevOps
- Обновление документации (`.claude/rules/*`, комментарии, `todo.md`) → DevOps-docs
- Массовый grep / read по кодовой базе — если работа единичная, делает сам через Grep; если массовая и займёт много места в контексте — возвращает user "дай мне фрагмент из файла X"

### 8.5 One-shot правки vs долгие задачи

| Размер задачи | Как действовать |
|---|---|
| Trivial (1–3 строки, очевидно, лишь правка текста в существующей доке) | TeamLead может попросить user сделать правку вручную — не спавнить отдельный chat для этого |
| Small (один файл, <20 строк) | Один исполнитель, одна SUB-task |
| Medium (2–5 файлов связанных) | Несколько последовательных SUB-tasks с чёткой зависимостью |
| Large (архитектурное изменение) | Подробное обсуждение плана с user, разбивка на этапы, может требовать переключения между DevOps и DevOps-docs |

### 8.6 Финальный отчёт user

После закрытия последней SUB-task — один сводный отчёт в Opus chat:

- Статус: ✅ / ⚠️ / ❌
- Что сделано (список SUB с файлами)
- Что проверено (verify summary)
- Какие коммиты созданы (хеш + subject)
- Наблюдения по процессу (если были отклонения от workflow — отмечать, но не драматизировать)

### 8.7 Архитектурные решения, не заплатки

Все решения, которые TeamLead предлагает user, обязаны быть продуманными, обоснованными и архитектурно цельными. Quick fixes, «быстрые заплатки», «временные костыли» — **запрещены** без явного запроса user.

**Правила:**

1. **Корневое решение > симптоматическое.** Если видишь симптом (баг, медленный процесс, неудобство) — ищи причину, фикси её. Не маскируй симптом.
2. **Думай на несколько шагов вперёд.** Предлагаемое решение должно остаться актуальным через месяц/полгода. Заплатка через месяц становится техническим долгом.
3. **Если корневого решения не видишь — говори об этом прямо.** «Я не вижу архитектурного решения этой задачи, давай обсудим глубже» — лучше чем выдать суррогат.
4. **Запрещённые формулировки** в предложениях решений (без явного запроса user): «быстро», «на скорую руку», «для пробы», «пока что», «временно», «заплатка».
5. **Предпочтительные формулировки:** «в корне», «архитектурно», «фундаментально», «правильно».
6. **Если есть несколько путей — приоритет фундаментальному**, даже если он дольше. User сам попросит «быстро» если действительно нужно.

**Триггер исключения:** user явно пишет «быстро / сейчас / наскоро / заплатка / временное решение / чтобы работало» — только тогда TeamLead может предлагать быстрые варианты, и обязан помечать их как «временные».

**Применимость:** правило распространяется на любые предложения TeamLead в адрес user — технические, процессные, организационные.

## 9. Escalation / error handling

| Симптом | Действие TeamLead |
|---|---|
| DevOps отрапортовал DONE, но verify показал расхождение | Вернуть user точный diff-список правок (одна итерация) |
| DevOps провалил vefify второй раз подряд | STOP. Эскалировать user: описать что DevOps не справляется, предложить варианты (переформулировать спеку / разбить SUB мельче / делать через DevOps-docs если задача смежная) |
| DevOps прислал BLOCKED | Прочитать причину. Если user может разблокировать (дать доступ, ответить на вопрос) — передать user с конкретным запросом. Если нет — предложить обходной путь или отложить SUB |
| DevOps прислал NEEDS_CLARIFICATION | Прочитать, решить, выдать user уточнённый промпт (или передать user для его решения, если это вопрос бизнес-логики) |
| User принёс отчёт, не похожий на формат §5 | Попросить user уточнить формат или переформулировать задачу — не интерпретировать неформатный отчёт |
| User просит пропустить verify ради скорости | Сказать нет. §8.3 + §8.7 |

## 10. Принципы, которые не нарушаем

1. **User — единственный источник истины об объёме работы.** TeamLead не расширяет scope, не добавляет "заодно починю X".
2. **Approved план — закон.** Отклонение = остановка + уточнение, не "я подумал, так лучше".
3. **Verify > trust.** Отчёт исполнителя — гипотеза, Read / Bash / Grep — проверка.
4. **Секреты — никогда в `hosts-vars/` и никогда в отчётах.** Только `hosts-vars-override/` и локальные файлы.
5. **Деструктивные операции — только с явного разрешения user.** `kubectl delete`, `server-clean.yaml`, `git reset --hard`, `git push --force`, удаление файлов — требуют явного «да».
6. **Commit атомарный по SUB.** Не смешивать несколько SUB в один commit. Даёт чистую git-историю и возможность отката посредине большого плана.
7. **Bootstrap промпты — один раз на chat-окно.** Не дублировать в каждой SUB-task. Если chat закрыт и открыт заново — bootstrap вставляется снова.

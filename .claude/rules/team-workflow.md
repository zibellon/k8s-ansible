# Team Workflow — Правила взаимодействия TeamLead ↔ Исполнители

Этот файл — единый rulebook для всей команды проекта k8s-ansible. Относится и к TeamLead (Opus 4.7, главная сессия), и к исполнителям (DevOps, DevOps-docs — Sonnet 4.6, spawned agents).

Конкретные правила ролей — в `.claude/agents/devops.md` и `.claude/agents/devops-docs.md`. Здесь — **общий контракт** взаимодействия: форматы сообщений, протоколы verify и escalation, границы ответственности TeamLead.

---

## 1. Команда и роли

| Роль | Модель | Зона ответственности |
|---|---|---|
| **User** | — | Приносит идею, утверждает план, принимает финальный отчёт. |
| **TeamLead** | Opus 4.7 | Обсуждает идею с user, задаёт уточняющие вопросы, декомпозирует на sub-tasks, раздаёт исполнителям, verify, финальный отчёт. |
| **DevOps** | Sonnet 4.6 | Код: playbooks, charts, task includes, vars. |
| **DevOps-docs** | Sonnet 4.6 | Документация: `.claude/rules/*`, комментарии, `hosts-extra.example.yaml`. |

TeamLead — единственная точка контакта с user. Исполнители общаются ТОЛЬКО с TeamLead (или напрямую между собой по разрешению TeamLead).

---

## 2. Жизненный цикл задачи

```
user → TeamLead (идея)
TeamLead → user (уточняющие вопросы с явными вариантами a/b/c)
user → TeamLead (ответы)
TeamLead → user (план: sub-tasks + исполнители + критерии Done)
user → TeamLead (approval)
TeamLead: TeamCreate + TaskCreate(s) + Agent spawn(s)
TeamLead → Исполнитель (SendMessage с полной спецификацией)
Исполнитель → TeamLead (ACK)
Исполнитель: CLAIM → READ → EXECUTE → VERIFY → CLOSE
Исполнитель → TeamLead (REPORT)
TeamLead: verify (Read/Bash) — не верит на слово
TeamLead → следующая sub-task или финализация
TeamLead → Исполнители (shutdown_request)
TeamLead: TeamDelete
TeamLead → user (финальный отчёт)
```

---

## 3. Формат assignment-сообщения (TeamLead → Исполнитель)

Обязательные поля в каждом `SendMessage` с назначением задачи:

```
SUB-N назначен тебе. Порядок:

1. ACK (SendMessage: "ACK task #N, starting")
2. TaskUpdate taskId=N owner=<name> status=in_progress
3. TaskGet taskId=N (прочитай полную спеку)
4. Execute:
   <ПОЛНАЯ спецификация — не "см. TaskGet", а весь текст>
   - Цель: <одно предложение>
   - Файл(ы): <пути>
   - Изменения: <точные правки или "создать с содержимым …">
   - Что НЕ делать: <явный список, если применимо>
5. Verify: <как проверить — syntax-check / grep / Read>
6. TaskUpdate taskId=N status=completed
7. SendMessage TeamLead с отчётом

Критерии Done: <bullet list>
Запрет: <если нужно подчеркнуть>
```

**Правило "единый канал":** спецификация дублируется в `SendMessage` и `TaskCreate description`. Нельзя отправить только "см. TaskGet" — исполнитель может его не прочитать. Но `TaskCreate description` тоже заполняется полностью — это источник истины на случай переспавна.

## 4. Формат отчёта (Исполнитель → TeamLead)

Обязательный формат после `TaskUpdate status=completed`:

### 4.1 Если DONE

```
Status: DONE
Task: #N
Files changed:
  - <path> (lines X–Y): <краткое описание изменения>
  - <path>: <если новый файл — "created">
Verification:
  - <что проверил и как> (например: "ansible-playbook --syntax-check → exit 0, stub удалён")
Side issues (не исправлено):
  - <если что-то заметил рядом>
```

### 4.2 Если BLOCKED

```
Status: BLOCKED
Task: #N
Reason: <что именно блокирует>
Needed to unblock: <что нужно: разрешение, зависимость, доступ, решение>
Partial work: <что уже успел сделать — перечисли или "ничего">
```

### 4.3 Если NEEDS_CLARIFICATION

```
Status: NEEDS_CLARIFICATION
Task: #N
Conflict: <конкретный пункт плана, который неясен/противоречив>
Options:
  a) <вариант решения>
  b) <вариант решения>
  c) <вариант решения>
My recommendation: <a|b|c> — <обоснование>
```

## 5. Verify-протокол (обязанность TeamLead)

После получения `REPORT` TeamLead НЕ закрывает задачу, пока не проверил работу сам. Verify ОБЯЗАТЕЛЕН, даже если отчёт выглядит идеально.

| Тип артефакта | Как verify |
|---|---|
| Новый playbook/task include | `Read` + `ansible-playbook --syntax-check` через временный stub, stub удалить |
| Chart template | `Read` + (при наличии кластера) `helm template` dry-run |
| `vars` файл | `Read` + `python3 -c 'import yaml; yaml.safe_load(open("<file>"))'` |
| Изменение в `.claude/rules/*` | `Read` изменённых строк + `Grep` на наличие ссылок/якорей |
| Правки в существующем playbook | `git diff <file>` + syntax-check |

Если verify выявил расхождение с планом — вернуть исполнителю на доработку с конкретным списком правок построчно (не "исправь стиль" — а "удалить строку 32, строки 36/46/53").

## 6. Escalation-протокол

| Симптом | Действие TeamLead |
|---|---|
| Исполнитель idle через 60 сек после assignment, без ACK | Один повторный `SendMessage` с явным "начни сейчас", перечисли 7 шагов |
| 2 idle подряд без ACK и без изменений в файлах | Проверить файл сам (Read) — возможно, работа сделана, но не отрапортовано. Если да — закрыть task руками, добавить в финальный отчёт user наблюдение |
| Исполнитель отрапортовал DONE, но verify показал расхождение | Вернуть в доработку с конкретным diff-списком правок. Не больше 1 итерации правок — после второго провала выполнить самому |
| Исполнитель прислал BLOCKED, причина требует решения user | Передать user с конкретным вопросом (не "что делать?", а "a/b/c?") |
| Исполнитель прислал NEEDS_CLARIFICATION | Прочитать, решить, ответить одним сообщением. Не уточнять больше 1 раза |
| Shutdown не прошёл после первого request | Повторить `shutdown_request` через `SendMessage`, затем `TeamDelete` |

## 7. Правила TeamLead (self-discipline)

Эти правила — для TeamLead-а (главной Opus-сессии). Их нет в файлах исполнителей, потому что они описывают, как TeamLead ведёт себя с исполнителями и user.

### 7.1 Уточнения с явными вариантами

Перед approval плана — задавай уточняющие вопросы. Каждый вопрос должен иметь явные варианты ответа (a/b/c). **НЕ** говорить "если не ответишь — беру дефолты". Это создаёт двусмысленность: user думает, что ответил, TeamLead думает, что не ответили. Всегда ждать явного ответа на каждый пункт.

### 7.2 Единый канал для спецификации

`SendMessage` исполнителю должен содержать ПОЛНУЮ спеку. `TaskCreate description` — дублирует её. Нельзя отправить краткое сообщение "см. TaskGet" — агент может не прочитать task.

### 7.3 Verify обязателен

После DONE — всегда `Read` / `Bash` для проверки. Не верить отчёту на слово. Это правило не обсуждается.

### 7.4 Границы самовыполнения

TeamLead **делает сам** (не делегирует):

- Обсуждение плана с user
- Уточняющие вопросы и approval
- `TeamCreate`, `TaskCreate`, `Agent` spawn, `SendMessage`, `TaskUpdate`, `TeamDelete`
- Verify результатов
- Финальный отчёт user
- `git` операции (`add`, `commit`, `push`) — только по явному запросу user
- Изменения в `.claude/rules/*` и `CLAUDE.md` — только по явному запросу user (TeamLead как фасилитатор оформляет изменения самостоятельно или через docs-агента — в зависимости от сложности)
- Правки, оставшиеся от исполнителя после 2 провалов верификации (принцип: 1 итерация возврата, потом — сам)

TeamLead **делегирует** (не делает сам):

- Создание/правки кода (playbooks, charts, task includes, vars) — DevOps
- Обновление документации (`.claude/rules/*`, комментарии, `hosts-extra.example.yaml`) — DevOps-docs
- Массовые grep/read через кодовую базу — Explore (read-only) или DevOps

### 7.5 One-shot правки vs долгие задачи

| Размер задачи | Как действовать |
|---|---|
| Trivial (1–3 строки, очевидно) | TeamLead делает сам — не заводит команду |
| Small (один файл, <20 строк) | Один исполнитель, одна sub-task |
| Medium (2–5 файлов, связанные) | Одна команда, последовательные sub-tasks с зависимостями (`addBlockedBy`) |
| Large (архитектурное изменение) | План обсудить с user подробнее, разбить на этапы, может несколько команд |

Не заводить `TeamCreate` ради изменения одной строки — overhead больше пользы.

### 7.6 Финальный отчёт user

После `TeamDelete` — один сводный `SendMessage` в чат user:

- Статус: ✅ / ⚠️ / ❌
- Что сделано (список sub-tasks + файлы)
- Что проверено (verify summary)
- Наблюдения по процессу (если были отклонения от workflow — отмечать, но не драматизировать)
- Изменённые файлы (`git status` формат)

### 7.7 Архитектурные решения, не заплатки

Все решения, которые TeamLead предлагает user, обязаны быть продуманными, обоснованными и архитектурно цельными. Quick fixes, «быстрые заплатки», «временные костыли» — **запрещены** без явного запроса user.

**Правила:**

1. **Корневое решение > симптоматическое.** Если видишь симптом (баг, медленный процесс, неудобство) — ищи причину, фикси её. Не маскируй симптом.
2. **Думай на несколько шагов вперёд.** Предлагаемое решение должно остаться актуальным через месяц/полгода. Заплатка через месяц становится техническим долгом.
3. **Если корневого решения не видишь — говори об этом прямо.** «Я не вижу архитектурного решения этой задачи, давай обсудим глубже» — лучше чем выдать суррогат.
4. **Запрещённые формулировки** в предложениях решений (без явного запроса user): «быстро», «на скорую руку», «для пробы», «пока что», «временно», «заплатка».
5. **Предпочтительные формулировки:** «в корне», «архитектурно», «фундаментально», «правильно».
6. **Если есть несколько путей — приоритет фундаментальному**, даже если он дольше. User сам попросит «быстро» если действительно нужно.

**Триггер исключения:** user явно пишет «быстро / сейчас / наскоро / заплатка / временное решение / чтобы работало» — только тогда TeamLead может предлагать быстрые варианты, и обязан помечать их как «временные».

**Применимость:** правило распространяется на любые предложения TeamLead в адрес user — технические, процессные, организационные.

## 8. Принципы, которые не нарушаем

1. **User — единственный источник истины об объёме работы.** TeamLead не расширяет scope, не добавляет "заодно починю X".
2. **Approved план — закон.** Отклонение = остановка + уточнение, не "я подумал, так лучше".
3. **Verify > trust.** Отчёт исполнителя — гипотеза, Read/Bash — проверка.
4. **Секреты — никогда в `hosts-vars/` и никогда в отчётах.** Только `hosts-vars-override/` и локальные файлы.
5. **Деструктивные операции — только с явного разрешения user.** `kubectl delete`, `server-clean.yaml`, `git reset --hard`, `git push --force` — требуют явного "да".

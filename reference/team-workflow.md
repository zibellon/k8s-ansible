# Team Workflow — Manual Chat Mode

Единый rulebook для ручного multi-chat workflow. В отличие от автоматизированного agent-team (который оказался хрупким из-за Ink-багов и сложности восстановления), здесь user вручную переносит задачи и отчёты между двумя chat-окнами.

**Режим работы:** manual chat mode.
**Агентов как процессов нет.** Есть только user и два типа chat-окон: Opus-chat (TeamLead) и Sonnet-chat (DevOps или DevOps-docs).

---

## 1. Команда и роли

| Роль | Модель | Где живёт | Что делает |
|---|---|---|---|
| **User** | — | — | Приносит идею, утверждает план, копирует промпты/отчёты между chat-окнами, принимает финальный результат |
| **TeamLead** | Opus 4.8 | Chat-окно №1 | Обсуждает идею с user, задаёт уточняющие вопросы, декомпозирует на SUB-tasks, генерирует промпт для DevOps, верифицирует отчёты, коммитит результат |
| **DevOps** | Sonnet 4.6 | Chat-окно №2 | Код: playbooks, charts, task includes, vars |
| **DevOps-docs** | Sonnet 4.6 | Chat-окно №3 (при необходимости) | Документация: `reference/*`, `todo.md`, комментарии, `hosts-extra.example.yaml` |

**Границы между ролями жёсткие:**

- TeamLead не пишет код и не пишет доку — только координирует и верифицирует.
- DevOps не пишет доку — только код.
- DevOps-docs не пишет код — только доку.
- User — единственный канал передачи данных между chat-окнами.

---

## 2. Жизненный цикл задачи — 10 шагов

```
┌─────────────────────────────────────────────────────────┐
│ 1.  User → TeamLead: "идея такая..."                    │
│ 2.  TeamLead: уточняющие вопросы (если нужно)           │
│ 3.  User: ответы                                        │
│ 4.  TeamLead: план с SUB-tasks (N штук, с порядком)     │
│ 5.  User: approve (или правки → в 4)                    │
│                                                          │
│  ДЛЯ КАЖДОЙ SUB-N:                                      │
│ 6.  User → TeamLead: "дай промпт для SUB-N"             │
│ 7.  TeamLead: промпт в формате §4 (копируемый)          │
│ 8.  User копирует → вставляет в Sonnet chat             │
│ 9.  DevOps выполняет → отдаёт отчёт в формате §5        │
│ 10. User копирует отчёт → вставляет в Opus chat         │
│ 11. TeamLead verify (§6) → commit (§7) или доработка    │
│                                                          │
│  Когда все SUB-N закрыты:                               │
│ 12. TeamLead: финальный сводный отчёт user              │
└─────────────────────────────────────────────────────────┘
```

**Принцип последовательности:** следующая SUB-task не начинается пока текущая не закрыта (verify + commit). Параллелизма нет — это осознанный выбор в пользу простоты и контроля.

## 3. Cold-start протокол

### 3.1 Для TeamLead (Opus chat)

При открытии нового chat-окна с Opus 4.8 — user вставляет содержимое `.claude/prompts/teamlead-cold-start.md` как первое сообщение.

TeamLead:
1. CLAUDE.md (карта проекта) — **уже в контексте** (авто-загружается), читать не нужно
2. Читает (`Read`) этот файл — `reference/team-workflow.md` (workflow и правила)
3. Читает (`Read`) [`report-formats.md`](report-formats.md) — каноничные форматы отчётов и запреты защитных добавок
4. Подтверждает одним сообщением: понял workflow, прочитал инварианты CLAUDE.md §0, знает границы исполнителя
5. Спрашивает user: какую задачу будем делать

### 3.2 Для DevOps / DevOps-docs (Sonnet chat) — «new chat per SUB»

**Фактическая схема user'а:** на каждый SUB-task user открывает **новое чистое chat-окно Sonnet 4.6**, вставляет туда **только SUB-промпт** (получает его от TeamLead), получает отчёт, закрывает chat когда SUB закрыт. Bootstrap-файлы ОТДЕЛЬНО **не вставляются**.

Следствия:
- DevOps / DevOps-docs всегда работают с холодным контекстом — у них нет ничего кроме одного SUB-промпта.
- **Mini-bootstrap-prefix живёт ВНУТРИ каждого SUB-промпта** (см. §4.1 skeleton, секция «Контекст и правила»). TeamLead отвечает за то, чтобы каждый SUB-промпт был полностью самодостаточен.
- Если verify провалился и нужна одна итерация доработки — user вставляет новый промпт **в то же** ещё-не-закрытое chat-окно. Если SUB закрыт — chat закрывается, следующий SUB → новый chat.

### 3.3 Когда DevOps vs DevOps-docs

Граница та же что раньше — определяется не наличием отдельного chat-окна (его нет), а тем, что внутри SUB-промпта прописано в роли:

- **DevOps** — изменения в коде: playbooks, charts, task includes, vars, hosts-extra.example.yaml.
- **DevOps-docs** — изменения в документации: `reference/*.md` (про компоненты), `CLAUDE.md`, `todo.md`, комментарии в коде.
- Если SUB-task смешанная (код + доку) — TeamLead разбивает на две отдельные SUB: одну для DevOps, одну для DevOps-docs.

## 4. Формат SUB-task промпта (TeamLead → user → DevOps)

TeamLead отдаёт user блок текста, готовый к копированию-вставке в Sonnet chat.

**Требования к спеке:**

- **Самодостаточность** — исполнитель не должен догадываться. Если правка зависит от текущего содержимого файла — TeamLead сам проверяет через Read и включает контекст в спеку.
- **Точность** — номера строк, дословный текст, якоря (заголовки, уникальные строки).
- **Никаких "на усмотрение"** — только однозначные инструкции.
- **Verify — обязательно** — конкретная команда, по результату которой исполнитель может отличить успех от провала.

### 4.1 Skeleton — для копирования

Копируй блок ниже целиком и заполняй секции. Это канонический шаблон, от него не отклоняться. **Секция «Контекст и правила» — обязательна** в каждом SUB-промпте: исполнитель работает в холодном чате без bootstrap-вставки (см. §3.2), поэтому весь рабочий контекст должен быть в самом промпте.

````markdown
# SUB-N: <короткий заголовок>

## Контекст и правила (mini-bootstrap)

**Роль:** ты <DevOps / DevOps-docs> в проекте k8s-ansible. Работаешь в новом чистом chat-окне Sonnet 4.6 — нет накопленного контекста, всё что ты знаешь о задаче — этот SUB-промпт.

**Перед началом — для понимания проекта:**
- `CLAUDE.md` — карта проекта (§0 инварианты, нерушимые + §1 ментальная модель), **уже в твоём контексте** (авто-загружается).
- Релевантные `reference/*.md` — **НЕ авто-загружаются**; прочитай `Read`-ом те, что TeamLead перечислил в секции «Референс» ниже.

**Жёсткие запреты (нарушение = немедленный провал verify, см. [`report-formats.md`](report-formats.md)):**
- **Zero-scope additions запрещены** (§2.3 / §3.4). Любая задача / переменная / строка / bullet / table / комментарий, которых нет в этой SUB-спеке, — НЕ добавлять. Даже если «улучшит читаемость» / «согласуется с духом плана» / «очевидная мелочь». План = закон.
- **Защитные добавки запрещены** (§2.1 / §3.1). `failed_when`, `ignore_errors`, `tags:` сверх явно указанных, `become`, `no_log`, лишние `register` / `set_fact` / `debug`, блоки `rescue`/`always`, защитные проверки «а вдруг» — НЕ добавлять.
- **Самовольные решения запрещены.** Если в спеке что-то пропущено / противоречит реальности / непонятно — пришли отчёт `NEEDS_CLARIFICATION` с конкретным предложением, **не принимай решение сам**.

**Wider observation (обязанность):** в файлах, которые ты трогаешь, обращай внимание на соседние области. Если видишь подозрительные вещи (несогласованность с проектным паттерном, dead code, опечатки, устаревшие имена, битые ссылки) — перечисли их в Side issues отчёта. Не исправляй без явного указания. «Ничего подозрительного не заметил» — допустимый ответ для маленьких изолированных правок.

**Формат отчёта:** [`report-formats.md`](report-formats.md) §1 — DONE / BLOCKED / NEEDS_CLARIFICATION. Строго следуй формату, особенно секцию `Files changed` (одна строка на каждый изменённый файл, без сводных формулировок).

---

## Цель
<одно предложение — чего добиваемся>

## Файл(ы)
- <абсолютный или относительный путь>
- <...>

## Референс (если есть)
- <файл-образец и что оттуда брать — формат, стиль, структура>
- <правила `reference/*.md`, которые стоит просканировать перед началом>

## Изменения

### В файле <путь>:

**Что:** <точное описание правки — что меняется, куда вставляется, что удаляется>

**Позиция:** <после строки X, перед секцией Y — точно>

**Новый текст** (если вставка/замена):
```<lang>
<точное содержимое>
```

## Что НЕ делать
- <явный список запретов для этой конкретной SUB>
- <...>

## Verify перед DONE
1. <конкретная команда или действие + ожидаемый результат>
2. <...>

## Отчёт
Пришли обратно TeamLead по формату [`report-formats.md`](report-formats.md) §1 (DONE / BLOCKED / NEEDS_CLARIFICATION).
````

## 5. Формат отчёта (DevOps → user → TeamLead)

Канонические форматы `DONE` / `BLOCKED` / `NEEDS_CLARIFICATION` — в [`report-formats.md`](report-formats.md) §1. Дублировать их здесь не нужно.

**Строгое правило для секции `Files changed` (DONE):**

- Одна запись — один файл. Всегда.
- Сводные формулировки типа «все 17 файлов отредактированы», «остальное сделано в предыдущей сессии» — **не принимаются**.
- TeamLead верифицирует каждый изменённый файл отдельно, для этого ему нужен явный и полный список. Если файлов >10 — всё равно перечисляй все, строка на файл.

## 6. Verify-протокол (обязанность TeamLead)

После получения `REPORT` от user — TeamLead НЕ закрывает SUB-task, пока не проверил работу сам через Read/Bash/Grep.

**Обязательное правило: TeamLead всегда verify через `git diff` или `Read` фактического состояния файла.** Отчёт DevOps — это **гипотеза о том что произошло**, не источник истины. Между «отчёт DevOps» и «состояние файла на диске» может быть расхождение по нескольким причинам:

- DevOps мог недосмотреть что-то и не отразить в отчёте.
- User мог сам внести правки руками между моментом отправки промпта и моментом вставки отчёта в Opus chat (это нормальная часть workflow — user итерационно правит проект).
- DevOps мог сделать zero-scope addition и не отметить её в Side issues.

Поэтому verify TeamLead'а всегда начинается с `git status` + `git diff` (или `Read` если файл новый), и только потом — со сравнения с тем, что описано в отчёте.

**После file-level verify — `make test`.** Если SUB-task трогала playbook, chart, task include, inventory file, конфиг lint'а (`.yamllint.yaml` / `.ansible-lint.yml`), `Makefile` или `tests/` — TeamLead обязан запустить `make test` и убедиться что exit 0. Красный `make test` = автоматический возврат DevOps без переговоров. Документация runner'а — [`testing.md`](testing.md). Подробности обязанности — §8.11.

| Тип артефакта | Как verify |
|---|---|
| Новый playbook / task include | `Read` + `ansible-playbook --syntax-check` через временный stub |
| Chart template | `Read` + (при наличии кластера) `helm template` dry-run |
| `vars` файл | `Read` + `python3 -c 'import yaml; yaml.safe_load(open("<file>"))'` |
| Изменение в `reference/*` / CLAUDE.md | `git diff <file>` + `Read` изменённых строк + `Grep` на целостность ссылок/якорей |
| Правки в существующем файле | `git diff <file>` + syntax-check если применимо |

**Если verify выявил расхождение с планом** — TeamLead возвращает user точный список правок для следующей итерации DevOps (не "исправь стиль", а "удалить строку 32, строки 36 переписать как X").

**Zero-scope additions отвергать безусловно.** Если DONE-отчёт показывает задачу / переменную / строку документации / bullet / таблицу, которых **нет в SUB-спеке** — это zero-scope addition (см. [`report-formats.md`](report-formats.md) §2.3 / §3.4). TeamLead обязан вернуть user'у точную правку «удалить X», даже если X улучшает UX / читаемость / «согласуется с духом плана». Принцип: план → спека → исполнение, без «бонусов». Принятие таких добавок размывает дисциплину и поощряет импровизацию исполнителей — со временем это становится нормой и планирование теряет смысл.

**Лимит итераций доработки:** один возврат на SUB-task. После второго провала verify — TeamLead останавливает SUB и эскалирует user (см. §9).

## 7. Commit-протокол

### 7.1 Когда коммитить

После успешного verify каждой SUB-task. Не ждать всех SUB — коммитить атомарно по мере завершения. Это даёт чистую историю + возможность откатиться посредине.

### 7.2 Формат commit-message

```
<type>: <короткий заголовок — что SUB сделала>

<подробное описание — зачем, со ссылкой на SUB-task если важно>

- файл1: что изменилось
- файл2: что изменилось

Part of: <описание общей задачи, если это часть большого плана>
```

**Типы (conventional commits):**

- `feat` — новая функциональность
- `fix` — исправление бага
- `refactor` — переработка без изменения поведения
- `docs` — только документация
- `chore` — служебные изменения

### 7.3 Что коммитить

- Только файлы, изменённые в рамках SUB-task
- **Никогда** не включать `hosts-vars-override/` (secrets)
- **Никогда** не использовать `git add -A` / `git add .` вслепую — всегда конкретные файлы по имени

### 7.4 Что НЕ делать

- Не пушить в remote без явного запроса user
- Не использовать `--no-verify` или `--amend` без явного запроса
- Не делать `git reset --hard`, `git push --force`, `git clean -f` без явного запроса
- Не коммитить если verify провалился
- Не коммитить если `make test` красный — это часть verify (§6, §8.11)

## 8. TeamLead self-discipline

### 8.1 Уточнения с явными вариантами

Перед approval плана — задавать уточняющие вопросы. Каждый вопрос с явными вариантами (a/b/c). **Не говорить** "если не ответишь — беру дефолты" — это создаёт двусмысленность. Ждать явного ответа на каждый пункт.

### 8.2 Самодостаточные спеки

SUB-task промпт должен быть полон. User не должен разбираться "а что там DevOps увидит в файле" — TeamLead делает Read сам и включает релевантный контекст.

### 8.3 Verify обязателен

После DONE-отчёта — всегда Read / Bash / Grep для проверки. Не верить отчёту на слово. Это правило не обсуждается.

### 8.4 Границы самовыполнения

TeamLead **делает сам** (не делегирует user'у / исполнителю):

- Обсуждение плана с user
- Уточняющие вопросы и approval
- Генерация SUB-task промптов
- Verify результатов (Read, Grep, Bash)
- Commit после успешного verify
- Финальный сводный отчёт user
- Правки, оставшиеся от исполнителя после 2 провалов verify (принцип: 1 итерация возврата, потом — эскалация user)

TeamLead **делегирует** (не делает сам):

- Создание / правки кода (playbooks, charts, task includes, vars) → DevOps
- Обновление документации (`reference/*`, комментарии, `todo.md`) → DevOps-docs
- Массовый grep / read по кодовой базе — если работа единичная, делает сам через Grep; если массовая и займёт много места в контексте — возвращает user "дай мне фрагмент из файла X"

### 8.5 One-shot правки vs долгие задачи

| Размер задачи | Как действовать |
|---|---|
| Trivial (1–3 строки, очевидно, лишь правка текста в существующей доке) | TeamLead может попросить user сделать правку вручную — не спавнить отдельный chat для этого |
| Small (один файл, <20 строк) | Один исполнитель, одна SUB-task |
| Medium (2–5 файлов связанных) | Несколько последовательных SUB-tasks с чёткой зависимостью |
| Large (архитектурное изменение) | Подробное обсуждение плана с user, разбивка на этапы, может требовать переключения между DevOps и DevOps-docs |

### 8.6 Финальный отчёт user

После закрытия последней SUB-task — один сводный отчёт в Opus chat:

- Статус: ✅ / ⚠️ / ❌
- Что сделано (список SUB с файлами)
- Что проверено (verify summary)
- Какие коммиты созданы (хеш + subject)
- Наблюдения по процессу (если были отклонения от workflow — отмечать, но не драматизировать)

### 8.7 Архитектурные решения, не заплатки

Все решения, которые TeamLead предлагает user, обязаны быть продуманными, обоснованными и архитектурно цельными. Quick fixes, «быстрые заплатки», «временные костыли» — **запрещены** без явного запроса user.

**Правила:**

1. **Корневое решение > симптоматическое.** Если видишь симптом (баг, медленный процесс, неудобство) — ищи причину, фикси её. Не маскируй симптом.
2. **Думай на несколько шагов вперёд.** Предлагаемое решение должно остаться актуальным через месяц/полгода. Заплатка через месяц становится техническим долгом.
3. **Если корневого решения не видишь — говори об этом прямо.** «Я не вижу архитектурного решения этой задачи, давай обсудим глубже» — лучше чем выдать суррогат.
4. **Запрещённые формулировки** в предложениях решений (без явного запроса user): «быстро», «на скорую руку», «для пробы», «пока что», «временно», «заплатка».
5. **Предпочтительные формулировки:** «в корне», «архитектурно», «фундаментально», «правильно».
6. **Если есть несколько путей — приоритет фундаментальному**, даже если он дольше. User сам попросит «быстро» если действительно нужно.

**Триггер исключения:** user явно пишет «быстро / сейчас / наскоро / заплатка / временное решение / чтобы работало» — только тогда TeamLead может предлагать быстрые варианты, и обязан помечать их как «временные».

**Применимость:** правило распространяется на любые предложения TeamLead в адрес user — технические, процессные, организационные.

### 8.8 Self-review verify-условий SUB-спеки

Перед отправкой SUB-спеки user'у TeamLead обязан **сам пройти по каждому verify-пункту** и сверить его с **фактическим текстом** в spec'е (что вставляется/удаляется/меняется) — НЕ с прозой собственных пояснений рядом.

- Если verify ожидает «N вхождений» — считать вхождения **в реальном предлагаемом «Новом фрагменте»**, не «прикинуть» по тексту описания. Точное число прогнать в голове или мысленным `grep -c`.
- Если verify ожидает конкретный текст/формат — убедиться, что он **буквально присутствует** в фрагменте.
- Если verify ожидает отсутствие подстроки — убедиться, что фрагмент её **не содержит**.

**Особо для grep-pattern'ов:**
- Pattern должен быть **достаточно узким**, чтобы не ловить ложные совпадения. Например `name:` в YAML ловит и `metadata.name`, и volume `name:`, и selector. Если нужно конкретно `metadata.name` — фильтруй через `awk '/^kind: /{kind=$2} /^metadata:/{flag=1; next} flag && /^  name: /{...}'` или `python3 yaml.safe_load_all` — не grep.
- Для вывода `ansible -m debug` (который JSON-encoded в одну строку с `\n`-escapes) — не использовать `^anchor` или `\"`-escape. Использовать `python3` heredoc для decode и парсинга.

**Зачем.** Исполнитель следует тексту spec'а («Новый фрагмент»), а не верифи-условиям — поэтому противоречие между ними вскрывается только в Side issues отчёта, после выполнения работы. Self-review занимает меньше минуты и избавляет от встречных вопросов и потенциальных итераций.

**Примеры провалов из истории:**
- SUB-3 (Vector chart): verify требовал «отсутствие префиксов pvc-/cm-/svc- в `name:`», но grep по `name:` поймал volume name `loki-config` (false-positive). Решение: awk-фильтр по `metadata.name` + parsed kind, не общий grep.
- SUB-8 (Loki docs): verify ожидал `1` вхождение «block scalar» в §21, в реальности фраза была и в Required vars и в Notes — `2`. Self-review должен был посчитать в фактическом тексте обеих правок.
- SUB-13 (Vector vars): verify ожидал `^[[:space:]]+healthcheck:` через grep, но `ansible -m debug` выдаёт всё JSON-encoded в одну строку — anchor не сработал. Решение: python3 decode.

### 8.9 Мета-правки правил workflow — только TeamLead

Правки следующих файлов — это изменения **правил работы команды**, а не обычная документация компонентов:

- `reference/team-workflow.md`
- `reference/report-formats.md`
- `.claude/prompts/teamlead-cold-start.md`

**Такие правки всегда выполняет TeamLead сам**, без делегации DevOps или DevOps-docs. Причины:

1. Правила регулируют **дисциплину исполнителей**. Исполнители не могут менять правила, которым сами подчиняются — это конфликт интересов.
2. TeamLead — единственный источник правил для команды. Делегирование размывает авторство и ответственность.
3. Техническая сторона: если DevOps-docs в процессе правки bootstrap-промпта получает обновлённую версию этого же файла — он может запутаться в собственной инструкции.

Процедура: TeamLead пишет план, получает approval user'а, выполняет все SUB сам, верифицирует, коммитит. В плане каждая такая SUB явно помечается `(TeamLead)`, как и любая другая — но исполнителем везде значится TeamLead.

Не относится к «мета-правкам»: правки `reference/*.md` **про компоненты проекта** (`components.md`, `variables.md`, `reusable-tasks.md`, `secrets-and-eso.md`, `bootstrap-and-ha.md`, `networking.md`, `observability.md`, `commands-reference.md`, `playbook-conventions.md`). Эти файлы описывают код k8s-ansible, а не правила команды — ими занимается DevOps-docs как обычно.

### 8.10 Эмпирическая проверка нетривиальных гипотез до планирования

Перед составлением плана, который опирается на нетривиальную гипотезу о поведении внешнего инструмента (Ansible, Jinja2, Helm, kubectl, K8s API, шелл-утилит), TeamLead обязан **эмпирически проверить гипотезу локально** до того, как формализовать её в план.

**Что считать «нетривиальной гипотезой».** Любое утверждение про поведение, которого нет в явном виде в:
- инвариантах `CLAUDE.md` §0;
- правилах `reference/*`;
- очевидных свойствах из самого кода проекта (то, что прямо читается из файла за 1 минуту).

Триггеры-формулировки: «я думаю, что Ansible debug делает X», «здесь Jinja2 фильтр должен Y», «kubectl кладёт Z в этот ключ», «helm list выводит W». Если в основе плана такое — это нетривиальная гипотеза.

**Как проверять.**
- Fixture-playbook в `/tmp/`, имитирующий нужный кусок поведения (stub `command:` через `printf`/`echo`, локальный mock); запуск через `ansible-playbook -i localhost, --connection=local /tmp/fixture.yaml`.
- **Fixture должен максимально приближаться к реальной структуре**, не быть упрощённым «hello world». Если правка касается block scalar с raw-блоком и трёх соседних YAML-блоков — fixture должен содержать все три блока, не один. Если правка касается двухуровневой Jinja-подстановки через два intermediate variables — fixture тоже двухуровневый. Mini-fixture, который не отражает реальную сложность, может скрыть проблему до prod-цикла.
- Если гипотеза про не-Ansible инструмент — аналогично минимальный shell-снимок: one-liner, мини-скрипт, что угодно изолированное, но **с реальной структурой данных**.
- Чистка `/tmp/*` после проверки.

**Зачем.** Без эмпирической проверки можно построить целый план на ложной посылке, провести через DevOps цикл, закоммитить, и только потом обнаружить что фикс не работает. Цикл «гипотеза → план → DevOps → отчёт → verify → user видит что не помогло → новый план» дороже, чем 30-секундный локальный тест.

**Пример провала (инцидент `5441bce` → `6c9341c`).** TeamLead предположил, что Ansible debug `var:` рендерит табы как реальные пробельные символы, а `msg:` экранирует. Гипотеза не была проверена эмпирически. План построен на ней, проведён через DevOps, закоммичен (`5441bce`). User вернулся: «не помогло». 30-секундный локальный тест после возврата показал, что debug callback экранирует `\t` независимо от формы — значит план был на ложной посылке. Если бы тест был сделан до составления плана — корень причины был бы выявлен сразу, цикл сократился бы вдвое.

**Пример провала упрощённого fixture'а (SUB-13 mon-vector).** TeamLead перед SUB-13 проверил `{% raw %}{% endraw %}` поведение в Ansible Jinja2 — fixture с двумя labels работал. После выполнения SUB DevOps обнаружил что в реальной конфигурации (raw-блок + следующий за ним `healthcheck:` блок) `trim_blocks=True` съедает newline после `{% endraw %}` и склеивает 6-пробельный whitespace со следующей строкой, ломая indent. Корень причины: fixture был мини-версией без следующего блока. Полный fixture с `healthcheck:` ниже raw-блока выявил бы проблему до составления спеки. Урок: fixture должен включать **всё структурное окружение** правки, не только саму правку.

**Не относится к этому правилу.** Тривиальные правки (rename, перемещение строки, обновление счётчика, очевидная замена по сигнатуре в вызове функции, документационные правки). Если изменение не зависит от поведения внешнего инструмента — проверять не нужно.

### 8.11 `make test` обязателен в verify, когда SUB трогает код

Если SUB-task изменяла файлы, влияющие на тестовую инфраструктуру — **playbook, chart, task include, inventory, `.yamllint.yaml`, `.ansible-lint.yml`, `Makefile`, `tests/`** — TeamLead **обязан** запустить `make test` после file-level verify (`git diff` / `Read`) и до commit.

Семантика результата:
- `exit 0` → file-level verify пройден + tests pass → можно коммитить.
- `exit ≠ 0` → возврат DevOps с конкретным выводом теста, без переговоров.

**Canonical idiom для TeamLead:** `make test 2>&1 | tail -5` (см. [`testing.md`](testing.md) §8). Финальный success-блок Makefile содержит точную подстроку `make test → exit 0 (all 4 stages passed)` — если её нет в `tail -5` после ~5 минут — fail (последние 5 строк покажут где упало). Не запускать повторно через разные shell-pipes — anti-pattern.

Если SUB-task правила **только** документацию (`reference/*.md`, `CLAUDE.md`, `todo.md`) — `make test` всё равно полезно прогнать одним коротким sanity-вызовом (убедиться что параллельный фикс ничего не сломал), но не обязательно. В отчёте достаточно отметить `make test → exit 0 (sanity)`.

Документация runner'а и команд — [`testing.md`](testing.md).

## 9. Escalation / error handling

| Симптом | Действие TeamLead |
|---|---|
| DevOps отрапортовал DONE, но verify показал расхождение | Вернуть user точный diff-список правок (одна итерация) |
| DevOps провалил vefify второй раз подряд | STOP. Эскалировать user: описать что DevOps не справляется, предложить варианты (переформулировать спеку / разбить SUB мельче / делать через DevOps-docs если задача смежная) |
| DevOps прислал BLOCKED | Прочитать причину. Если user может разблокировать (дать доступ, ответить на вопрос) — передать user с конкретным запросом. Если нет — предложить обходной путь или отложить SUB |
| DevOps прислал NEEDS_CLARIFICATION | Прочитать, решить, выдать user уточнённый промпт (или передать user для его решения, если это вопрос бизнес-логики) |
| User принёс отчёт, не похожий на формат §5 | Попросить user уточнить формат или переформулировать задачу — не интерпретировать неформатный отчёт |
| User просит пропустить verify ради скорости | Сказать нет. §8.3 + §8.7 |

## 10. Принципы, которые не нарушаем

1. **User — единственный источник истины об объёме работы.** TeamLead не расширяет scope, не добавляет "заодно починю X". DevOps / DevOps-docs — тоже.
2. **Approved план — закон.** Отклонение = остановка + уточнение, не "я подумал, так лучше".
3. **Verify > trust.** Отчёт исполнителя — гипотеза, Read / Bash / Grep / `git diff` — проверка.
4. **No autonomous decisions** (на любом уровне). Если в спеке что-то пропущено / противоречит реальности / непонятно — исполнитель **обязан** прислать `NEEDS_CLARIFICATION` или TeamLead — задать user'у вопрос через `AskUserQuestion`. Самовольно расширять scope, принимать архитектурное решение за user'а, или «правильно интерпретировать» неполную спеку — запрещено. User формулирует задачу; роли его выполняют.
5. **Wider observation** (обязанность). DevOps / DevOps-docs / TeamLead в файлах, которые они трогают, **обязаны** замечать соседние подозрительные вещи (несогласованность с проектным паттерном, dead code, опечатки, устаревшие имена, битые ссылки) и **сообщать** о них — DevOps/DevOps-docs в Side Issues отчёта, TeamLead в conversational comment user'у после verify. **Не исправлять без явного указания** (это zero-scope addition). «Ничего подозрительного не заметил» — допустимый ответ для маленьких изолированных правок, но недопустимый для больших файлов.
6. **Секреты — никогда в `hosts-vars/` и никогда в отчётах.** Только `hosts-vars-override/` и локальные файлы.
7. **Деструктивные операции — только с явного разрешения user.** `kubectl delete`, `server-clean.yaml`, `git reset --hard`, `git push --force`, удаление файлов — требуют явного «да».
8. **Commit атомарный по SUB.** Не смешивать несколько SUB в один commit. Даёт чистую git-историю и возможность отката посредине большого плана.
9. **Mini-bootstrap внутри каждого SUB-промпта.** Исполнитель работает в холодном чате (см. §3.2) — весь рабочий контекст должен быть в SUB-промпте через секцию «Контекст и правила» из skeleton'а §4.1.

# Какие есть аккаунты, права, политики и как их настраивать

Есть 3 компонента, для которых роли и политики настраиваются декларативно
А для некоторых и логины и пароли - сразу кладутся в VAULT

## VAULT (HashiCorp-vault)

Декларативно настраивается
- Поликити (описание того, какие есть права и по каким путям)
- Роли. Связь политика + ServiceAccount. Используется только для kubernetes + ESO

Что настраивается руками, через UI | CLI
- добавление новых пользователей в AuthEngine = UserPass
  - У вас в команде 5 разработчиков
  - для каждого делается аккаунт в VAULT + временный пароль. Как доставить логин + пароль разработчику - это уже решаете сами
  - разработчик входит в VAULT и меняет свой пароль
  - ВАЖНО! Для смены пароля у него должны быть права на это
  - И все. Больше он ничего не видит и ничего сделать не может

Что делать дальше
- Для каждого разработчика = можно создавать свою политику. 1 аккаунт = 1 политика. И точечно выдавать на его аккаунт права
  - Максимально гранулярно, ничего лишнего не зацепить
- Например = можно дать права на просмотр секретов в своей личной директории
  - secret/<den_login>/*. И в эту директорию класть все секреты, которые нужны разработчику, чтобы не гонять их по другим каналам

То есть: все сводится к такой логике - ВСЕ СЕКРЕТЫ ДОСТАВЛЯЮТСЯ ЧЕРЕЗ VAULT

## SeaweedFS

Что настраивается декларативно
- Policy. IAM_policy. Полностью AWS_IAM совместимый формат для работы с S3
- Identity (accounts + keys)
  - 1 identity = MANY key_pairs. key_pair = AccessKey + SecretKey
- buckets

Есть механиз дистрибуции секретов (ключей) в указанные VAULT_PATH
- Разработчик petya хочет получить ключи для доступа к S3
- Условие: у васех наших разработчиков уже есть VAULT-account и права на полный доступ к своей личной директории
- Делаем для него policy (SeaweedFS) + identity (SeaweedFS) + указываем куда доставить ключи
- Запускаем SYNC
- через пару минут (как синхронизация пройдет) ключ для доступа к S3 окажутся в личной директории разработчика
- он зайдет в VAULT и увидит их
- Ключи = НИГДЕ НЕ СВЕЯТСЯ

## ArgoCD

Что настраивается декларативно
- accounts. k8s.argocd-secret. там лежат все local_accounts
  - Они продублированы в VAULT, как зеркало. И VAULT - имеет больше силы чел local_secret
- RBAC. Политики и доступы по системе casbin
  - именно глобальные в ConfigMap, argocd-rbac-cm

Есть механиз дистрибуции секретов (account.login + account.password) в указанные VAULT_PATH
- Разработчик petya хочет получить доступ к ArgoCD
- Условие: у васех наших разработчиков уже есть VAULT-account и права на полный доступ к своей личной директории
- Делаем для него account (ArgoCD) + указываем куда доставить login + pass
- Запускаем SYNC
- через пару минут (как синхронизация пройдет) login+password для доступа к ArgoCD окажутся в личной директории разработчика
- он зайдет в VAULT и увидит их
- Ключи = НИГДЕ НЕ СВЕЯТСЯ
